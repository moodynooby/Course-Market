## 2025-05-15 - [Optimize Schedule Generation & Scoring]
**Learning:** Combinatorial search using generators with array spreading ([...current, section]) and recursive slicing ([first, ...rest]) causes excessive memory allocation and garbage collection overhead. Furthermore, performing scoring calculations (like Set creation and time parsing) inside the hot loop for every combination is a major bottleneck.
**Action:** Use an index-based backtracking approach with a single shared array (push/pop) to minimize allocations. Implement early pruning (e.g., credit limit checks) to skip invalid branches of the search tree. Hoist preference processing into a reusable `ScoringContext` to avoid redundant work in hot paths. Use Record-based lookups instead of `Array.indexOf` for O(1) performance.
